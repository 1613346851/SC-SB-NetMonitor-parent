# 网关服务（gateway-service）具体实现框架与文件结构
网关服务是基于**Spring Cloud Gateway 2021.0.8**开发的标准Spring Boot微服务，核心基于**WebFlux响应式框架**（需关闭Spring MVC自动配置），无任何持久化依赖，整体遵循**“配置驱动+过滤器链+内存缓存+跨服务REST调用”** 核心逻辑，按**接入层、采集层、处置层**做职责拆分，配合公共组件实现流量采集、防御执行、跨服务交互，所有层通过**全局过滤器链、线程安全内存缓存、REST客户端**完成联动，以下从**整体框架、各层具体实现（类/逻辑/作用）、公共核心组件、标准文件结构**四部分详细展开。

## 一、整体框架核心设计
1. 基础依赖：仅引入Spring Cloud Gateway核心包、Spring Boot WebFlux、RestTemplate、lombok（简化开发），无MyBatis/MySQL等持久化依赖；
2. 核心执行链路：**外部请求→接入层路由规则匹配→采集层全局过滤器（先采集）→处置层全局过滤器（后防御）→路由转发至靶场服务**，过滤器按**优先级排序**保证执行顺序；
3. 跨服务交互：通过**RestTemplate实现同步/异步REST调用**，完成与monitor-service的双向数据传输（推送流量DTO、接收防御指令、同步防御日志）；
4. 内存管理：基于**ConcurrentHashMap**实现线程安全的临时缓存，封装通用缓存工具类处理过期、清理逻辑，避免高并发下的线程安全问题；
5. 轻量化设计：无复杂中间件，所有配置通过yml文件+自定义配置类实现，符合毕设的轻量化架构要求。

## 二、各分层具体实现（类/逻辑结构/作用/实现方式）
### （一）接入层：路由规则配置层，无业务类，纯配置+原生组件驱动
接入层是网关的**请求入口规则层**，核心作用是实现**请求路由匹配与转发**，仅通过**配置文件+Spring Cloud Gateway原生组件**实现，无需开发自定义业务类，是网关最基础的分层。
1. **核心组件/类**
    - Spring Cloud Gateway原生自动装配类：`RouteLocator`（路由定位器，自动加载yml路由规则）、`GatewayProperties`（路由属性配置类，绑定yml配置）；
    - 自定义可选配置类：`GatewayRouteConfig`（路由自定义配置类，毕设场景可省略，纯yml配置即可）；
    - 核心配置文件：`application.yml`（网关核心配置，存储路由规则）。
2. **逻辑结构与作用**
    - 以**路由断言（Predicate）** 为核心，匹配`/target/**`前缀的所有请求，作为靶场服务的专属路由；
    - 配置**路由目标（URI）** 指向靶场服务（target-service）的部署地址，实现请求单向转发；
    - 配置**过滤器优先级**，指定全局过滤器（采集/防御）的执行顺序，同时关闭局部过滤器，保证全局过滤器链的唯一性；
    - 配置网关自身基础属性（端口、跨域、超时时间），避免请求转发超时影响采集逻辑。
3. **实现方式**
    - 在`application.yml`中配置**Path断言**：匹配`/target/**`请求路径，是靶场服务的唯一路由规则；
    - 配置**URI属性**：填写靶场服务的实际访问地址（如`http://localhost:9001`，毕设用固定地址，无需服务发现）；
    - 关闭Spring MVC自动配置（`spring.main.web-application-type=reactive`），适配WebFlux响应式框架；
    - 配置网关跨域规则，解决前端页面内嵌测试用例的跨域请求问题；
    - 若需扩展，可通过`GatewayRouteConfig`自定义动态路由，但毕设场景**纯yml静态配置**即可满足需求，降低复杂度。

### （二）采集层：网关核心业务层，基于全局过滤器实现全量流量采集与推送
采集层是网关服务的**核心数据层**，核心作用是**捕获全量请求信息、预处理、封装DTO、异步推送至monitor-service**，同时将流量数据存入临时缓存，所有逻辑基于**全局过滤器**实现，遵循“**先采集后转发**”原则，过滤器优先级高于处置层，保证流量数据无遗漏。
1. **核心组件/类（按职责拆分）**
    - **核心过滤器**：`TrafficCollectGlobalFilter`（实现`GlobalFilter+Ordered`，采集层全局过滤器，核心执行类）；
    - **工具类**：`ServerWebExchangeUtil`（请求信息提取工具）、`TrafficPreProcessUtil`（流量预处理工具）；
    - **数据载体**：`TrafficMonitorDTO`（跨服务流量DTO，核心）、`RawTrafficBO`（网关内部原始流量业务对象，过渡用）；
    - **跨服务客户端**：`MonitorServiceTrafficClient`（基于RestTemplate的流量推送客户端，专用于向monitor-service推送TrafficMonitorDTO）；
    - **配置类**：`RestTemplateConfig`（RestTemplate配置类，注入容器供客户端调用）。
2. **逻辑结构与作用**
    1. `TrafficCollectGlobalFilter`作为核心入口，实现`GlobalFilter`的`filter`方法和`Ordered`的`getOrder`方法，设置**最高优先级**，保证在所有过滤器中最先执行；
    2. 通过`ServerWebExchangeUtil`从`ServerWebExchange`（网关请求上下文）中**提取全量核心字段**：源IP、请求方法、URI、查询参数、请求体、目标服务IP、时间戳等，封装为网关内部的`RawTrafficBO`；
    3. `TrafficPreProcessUtil`对`RawTrafficBO`做**标准化预处理**：过滤健康检查、空请求等无效流量；对响应时间>3s、请求体大小>100KB的流量标记为“异常流量”；对请求体/查询参数做结构化处理，便于monitor-service分析；
    4. 将预处理后的`RawTrafficBO`转换为**跨服务标准DTO（TrafficMonitorDTO）**，完成网关与监测服务的数格式统一；
    5. `MonitorServiceTrafficClient`通过**异步RestTemplate调用**，将`TrafficMonitorDTO`推送至monitor-service的流量接收接口，同时将DTO存入**流量临时缓存**（近1小时）；
    6. 调用`chain.filter(exchange)`继续执行过滤器链，将请求转发至靶场服务，**采集逻辑为非阻塞异步执行**，避免影响请求转发效率。
3. **实现方式**
    - 利用WebFlux的**响应式特性**，通过`Mono/Void`实现异步采集与推送，避免同步调用阻塞网关线程；
    - 工具类抽离通用方法（字段提取、预处理），保证过滤器逻辑简洁，符合“单一职责”原则；
    - 源IP提取做兼容处理：适配浏览器/Postman/前端JS发起的请求，从请求头/请求对象中精准获取真实源IP；
    - 请求体提取通过`ServerWebExchange`的`getRequestBody()`实现，做流式读取，避免大请求体内存溢出；
    - 跨服务推送做**失败重试轻量处理**（毕设场景1次重试即可），保证流量数据尽可能同步至监测服务。

### （三）处置层：防御执行层，基于全局过滤器+REST接口实现防御联动
处置层是网关的**防御执行层**，核心作用是**接收monitor-service的高危攻击防御指令、执行轻量化防御操作、记录防御日志并同步至监测服务**，所有防御逻辑基于**全局过滤器**实现，过滤器优先级**低于采集层**，保证“先采集后防御”，同时提供REST接口供监测服务主动推送防御指令。
1. **核心组件/类（按职责拆分）**
    - **核心过滤器（按防御优先级排序）**：`IpBlacklistDefenseFilter`（IP拉黑过滤器）、`RequestRateLimitFilter`（请求限流过滤器）、`MaliciousRequestBlockFilter`（恶意请求拦截过滤器），均实现`GlobalFilter+Ordered`；
    - **防御指令接收**：`DefenseCommandController`（REST接口控制器，供monitor-service推送高危攻击防御指令）；
    - **数据载体**：`DefenseCommandDTO`（监测服务推送的防御指令DTO）、`DefenseLogDTO`（防御日志DTO，同步至监测服务）、`DefenseResultBO`（网关内部防御结果业务对象）；
    - **工具类**：`DefenseResponseUtil`（防御响应构建工具）、`DefenseLogUtil`（防御日志封装工具）；
    - **跨服务客户端**：`MonitorServiceDefenseClient`（基于RestTemplate的防御日志推送客户端，向monitor-service同步防御日志）。
2. **逻辑结构与作用**
   ##### 第一步：接收监测服务的防御指令
    - `DefenseCommandController`提供**POST REST接口**，接收monitor-service推送的`DefenseCommandDTO`，包含核心字段：源IP、防御类型（拉黑/限流/拦截）、防御过期时间、监测事件ID、风险等级等；
    - 接收到指令后，将信息**更新至对应的内存缓存**（黑名单缓存/限流缓存），为过滤器提供防御判定依据。

   ##### 第二步：过滤器按优先级执行防御操作（拉黑→限流→恶意请求拦截）
   三个防御过滤器按**优先级从高到低**执行，均通过`ServerWebExchange`获取请求源IP，查询内存缓存做防御判定，若触发防御则**构建响应对象并终止过滤器链**，不再转发请求至靶场服务：
    1. `IpBlacklistDefenseFilter`：检查源IP是否在**黑名单缓存**中，且未过期，若是则构建403禁止响应，终止请求；
    2. `RequestRateLimitFilter`：基于**限流缓存**的1秒固定窗口计数，统计同一IP的请求数，超阈值（默认10次/秒）则构建429限流响应，终止请求；
    3. `MaliciousRequestBlockFilter`：检查请求是否被监测服务标记为“恶意请求”，若是则构建自定义告警页面（含监测事件ID），终止请求。

   ##### 第三步：防御日志记录与同步
    - 所有防御操作均通过`DefenseLogUtil`封装为**跨服务标准DTO（DefenseLogDTO）**，包含字段：防御类型、目标IP、监测事件ID、触发原因、执行时间、响应状态码等；
    - `MonitorServiceDefenseClient`通过RestTemplate将`DefenseLogDTO`**异步推送至monitor-service**的防御日志接收接口，完成防御行为的持久化记录；
    - 防御日志推送为**异步非阻塞执行**，避免影响网关的防御响应效率。

3. **实现方式**
    - 过滤器通过`ServerWebExchange.getResponse()`构建**自定义响应对象**，设置响应状态码（403/429）、响应体（JSON/自定义页面），并通过`response.setComplete()`终止过滤器链，实现请求拦截；
    - 防御指令接口做**接口鉴权轻量处理**（毕设场景可通过请求头token简单鉴权），避免非法请求修改防御缓存；
    - 限流计数基于**时间窗口+ConcurrentHashMap**实现，按“IP+秒级时间戳”作为key，实现1秒窗口的精准计数；
    - 自定义告警页面通过**静态HTML字符串**构建，无需引入前端模板，保证网关轻量化。

## 三、临时缓存层：公共内存缓存组件，为采集层/处置层提供数据支撑
临时缓存层不属于独立业务分层，是**网关的公共基础组件**，为采集层、处置层提供**线程安全的临时数据存储**，所有缓存均基于**Spring内置ConcurrentHashMap**实现，封装通用缓存工具类处理**过期时间、定时清理、增删改查**，避免高并发下的线程安全问题和内存溢出。
1. **核心组件/类**
    - **通用缓存工具类**：`LocalCacheUtil`（封装ConcurrentHashMap的通用操作，含过期时间检查、定时清理、批量删除）；
    - **业务缓存类**：`TrafficTempCache`（近1小时流量缓存）、`IpBlacklistCache`（IP黑名单缓存）、`RequestRateLimitCache`（请求限流缓存），均基于`LocalCacheUtil`实现；
    - **常量类**：`GatewayCacheConstant`（缓存常量，定义缓存过期时间、缓存键前缀、清理频率）。
2. **逻辑结构与作用**
    - 所有业务缓存类为**单例设计**，注入Spring容器，供采集层/处置层的过滤器、控制器、工具类调用；
    - `TrafficTempCache`：存储近1小时的`TrafficMonitorDTO`，按“时间戳+IP”为key，定时清理过期数据，供网关实时流量查询；
    - `IpBlacklistCache`：存储黑名单IP与过期时间戳，键为源IP，值为过期时间戳，过滤器每次查询均做**过期自动检查**，避免无效黑名单；
    - `RequestRateLimitCache`：存储同一IP的秒级请求计数，键为“IP+秒级时间戳”，值为请求数，1秒窗口自动重置；
    - `LocalCacheUtil`通过**定时任务（ScheduledExecutorService）** 实现缓存的定时清理（如每分钟清理一次过期数据），避免内存溢出。
3. **实现方式**
    - 利用ConcurrentHashMap的**线程安全特性**，适配网关高并发请求场景，避免多线程下的缓存数据混乱；
    - 所有缓存操作均封装为**通用方法**（get/set/remove/expire），业务缓存类仅需注入`LocalCacheUtil`，无需重复实现；
    - 缓存过期时间通过`GatewayCacheConstant`全局定义（如黑名单10分钟、流量缓存1小时、限流1秒），便于统一修改；
    - 毕设场景为轻量化，缓存**仅在网关内存中生效**，无需做分布式缓存，符合架构设计要求。

## 四、网关服务公共核心组件
为保证代码解耦、可维护性，网关服务抽离**公共核心组件**，供所有分层调用，是网关的基础支撑，无业务逻辑，仅做通用能力提供。
1. **常量包（constant）**：`GatewayRouteConstant`（路由常量，如靶场路由前缀`/target/**`）、`GatewayCacheConstant`（缓存常量）、`GatewayFilterOrderConstant`（过滤器优先级常量）、`GatewayHttpConstant`（HTTP常量，如响应状态码、请求头）；
2. **异常包（exception）**：`GatewayBizException`（网关自定义业务异常）、`GatewayRequestException`（网关请求异常），统一异常类型；
3. **全局异常处理（handler）**：`GatewayGlobalExceptionHandler`（实现`ErrorWebExceptionHandler`，处理网关过滤器、接口的所有异常，返回标准化错误响应，避免网关直接抛出原生异常）；
4. **跨域配置（config）**：`CorsConfig`（网关跨域配置类，解决前端页面发起的跨域请求问题）；
5. **Lombok注解**：所有DTO/BO均使用`@Data/@NoArgsConstructor/@AllArgsConstructor`，简化实体类开发。

## 五、网关服务标准文件结构（Maven规范）
网关服务按**Maven标准目录结构**组织，所有代码在`src/main/java`下按**包职责**拆分，配置文件在`src/main/resources`下，测试包（`src/test/java`）毕设场景可省略，整体结构清晰，符合微服务开发规范，具体结构如下（按包/文件层级排列）：
