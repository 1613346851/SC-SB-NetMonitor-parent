# 网络监测系统 - 主动漏洞扫描功能设计
## 一、设计概述
本设计基于现有网络监测系统架构（靶场服务9001、网关服务9000、监测服务9002），在**完全保留现有漏洞监测功能**的基础上，扩展实现主动漏洞扫描功能，同时预留AI大模型接口扩展能力，支持手动预设漏洞与自动扫描漏洞双模式管理，不破坏现有代码结构、数据流向及业务逻辑，确保系统兼容性与可扩展性，贴合毕设重点（监测系统）的核心定位。

## 二、现有功能保留与兼容说明
### 2.1 保留全部现有功能
完全保留已实现的漏洞监测模块所有功能，包括但不限于：
1. 前端漏洞监测页面（vulnerability-monitor.html）：筛选栏（漏洞名称、风险等级、验证状态）、数据表格（ID、漏洞名称等字段）、详情弹窗、分页组件及JS交互逻辑（表格初始化、筛选、详情查看等）。
2. 后端核心代码：VulnerabilityMonitorController所有API接口、VulnerabilityVerifyServiceImpl漏洞验证逻辑、VulnerabilityStatServiceImpl统计逻辑、VulnerabilityMonitorEntity实体类及对应Mapper接口。
3. 联动逻辑：与流量接收模块、防御决策模块的联动，内存缓存机制、异步统计同步、攻击置信度提升等核心设计。

### 2.2 兼容要求
主动扫描功能新增的漏洞数据，需无缝接入现有漏洞管理体系，自动写入sys_vulnerability_monitor表，扫描结果可直接在现有漏洞监测页面展示、筛选、查看详情，完全复用现有统计、联动逻辑，无需修改原有代码。

## 三、主动漏洞扫描功能设计（基础版，无AI）
### 3.1 功能定位
主动漏洞扫描是监测服务（9002）的扩展功能，核心作用是主动对靶场服务（9001）发起安全探测，自动识别常见Web漏洞（SQL注入、XSS、路径遍历、命令注入等），并自动入库，与手动预设漏洞统一管理，提升系统主动安全监测能力。

### 3.2 架构分工
1. 靶场服务（9001）：作为被扫描目标，提供含漏洞的Web接口，响应监测服务发起的扫描探测请求，不参与扫描逻辑。
2. 网关服务（9000）：转发监测服务的扫描流量至靶场服务，统一流量入口，记录扫描过程中的流量数据，同步至流量监测表。
3. 监测服务（9002）：核心扫描引擎，负责扫描任务管理、漏洞探测、结果分析、自动入库，同时预留AI调用接口。

### 3.3 核心流程
1. 扫描启动：管理员通过新增的“漏洞扫描页面”，点击“开始扫描”，指定扫描目标（固定为靶场服务127.0.0.1:9001），选择扫描类型（全量扫描/快速扫描）。
2. 接口探测：监测服务自动探测靶场服务在线状态，爬取靶场所有显性Web接口（如/api/login、/api/search等），生成待扫描接口列表。
3. 漏洞探测：监测服务通过网关转发，向每个待扫描接口发送预设规则的测试Payload（SQL注入、XSS等对应测试语句）。
4. 结果研判：根据靶场服务的响应内容，通过规则匹配判断是否存在漏洞（如响应包含数据库错误则判定为SQL注入，包含脚本执行内容则判定为XSS）。
5. 自动入库：将扫描发现的漏洞自动写入sys_vulnerability_monitor表，默认标记verify_status=1（已验证可利用），标注漏洞来源为“自动扫描”，完善漏洞名称、类型、等级、路径、描述等信息。
6. 结果展示：扫描完成后，在“漏洞扫描页面”展示本次扫描结果，同时扫描发现的漏洞自动同步至现有“漏洞监测页面”，与手动预设漏洞统一展示、管理。

### 3.4 后端新增组件（扩展式开发）
#### 3.4.1 控制器层：VulnScanController
提供扫描相关API接口，供前端调用，不影响现有控制器逻辑：
| 接口 | 方法 | 功能 |
|------|------|------|
| `/api/vuln/scan/start` | POST | 启动漏洞扫描任务 |
| `/api/vuln/scan/progress` | GET | 获取扫描进度 |
| `/api/vuln/scan/result` | GET | 获取本次扫描结果 |
| `/api/vuln/scan/sync` | POST | 手动同步扫描结果至漏洞库（默认自动同步） |

#### 3.4.2 服务层：VulnScanService
核心扫描逻辑实现，包含接口探测、Payload发送、漏洞研判、自动入库等功能，调用现有VulnerabilityMonitorMapper插入漏洞数据，确保与现有系统兼容：
| 方法 | 功能 |
|------|------|
| `startScan()` | 启动扫描任务，初始化扫描参数，触发接口探测 |
| `detectInterfaces()` | 探测靶场服务接口，生成待扫描接口列表 |
| `sendPayloadAndDetect()` | 发送测试Payload，接收响应并研判漏洞 |
| `autoInsertVuln()` | 将扫描发现的漏洞自动插入现有漏洞表 |
| `getScanProgress()` | 实时返回扫描进度（已扫描接口数/总接口数） |

#### 3.4.3 辅助组件：Payload工具类
按漏洞类型封装固定测试Payload，供扫描服务调用：
- SqlInjectionPayload：SQL注入测试语句（如?id=1' or 1=1--）
- XssPayload：XSS测试语句（如<script>alert(1)</script>）
- PathTraversalPayload：路径遍历测试语句（如../../etc/passwd）
- CommandInjectionPayload：命令注入测试语句（如?cmd=whoami）

### 3.5 前端新增页面：漏洞扫描页面
基于Thymeleaf模板开发，与现有漏洞监测页面风格统一，核心功能：
1. 扫描配置：显示扫描目标（默认127.0.0.1:9001，不可修改），提供扫描类型选择（全量/快速）。
2. 任务控制：一键启动、暂停、终止扫描任务。
3. 进度展示：实时显示扫描进度条、已扫描接口数、已发现漏洞数。
4. 结果展示：扫描完成后，展示本次发现的漏洞列表（漏洞名称、类型、等级、路径），支持查看详情、一键同步至漏洞库。
5. 历史记录：展示过往扫描任务的时间、扫描结果、漏洞数量等信息。

## 四、AI大模型扩展设计（全局预留版 · 适配现有系统配置）
### 4.1 设计定位
AI大模型作为**全局智能辅助能力**接入系统，不局限于漏洞扫描模块，未来可扩展至流量分析、攻击识别、告警研判、安全决策等全场景监测功能。
本阶段基于**已搭建完成的监测服务框架与系统配置页面**，完成AI基础配置管理与统一调用入口预留，实现“一次配置、全局复用”，不影响现有基础功能，支持后续平滑扩展全栈AI能力。

### 4.2 前端实现：复用现有系统配置页面
无需新增独立AI配置页面，直接在**已完成的系统配置页面**中扩展AI相关配置项，完全复用现有增删改查、表单弹窗、数据回显、保存逻辑：
1. 配置项：在系统配置中新增两项AI核心参数
    - ai.model.url：第三方大模型接口地址
    - ai.model.apiKey：第三方大模型调用密钥
2. 交互方式：沿用现有配置编辑弹窗，无需新增页面与样式，保持系统界面统一。
3. 全局生效：配置保存后，漏洞扫描、流量分析、攻击检测、告警模块均可共用同一套AI配置，实现统一管理。

### 4.3 后端实现：全局AI预留服务
#### 4.3.1 数据库配置
直接使用现有`sys_config`表存储AI配置，无需新建表结构：
| 配置键 | 配置值说明 |
|--------|------------|
| ai.model.url | 全局AI大模型接口地址 |
| ai.model.apiKey | 全局AI大模型API密钥 |

#### 4.3.2 服务层：全局AiModelService（空实现，全局预留）
设计为**系统全局公共服务**，所有模块均可调用，不与扫描模块强绑定：
| 方法 | 功能（全局预留） |
|------|--------------|
| `generatePayload(...)` | 智能生成扫描Payload（供扫描模块使用） |
| `analyzeVuln(...)` | 漏洞智能研判（供扫描/监测模块使用） |
| `analyzeAttack(...)` | 攻击行为分析（供流量监测模块使用） |
| `generateFixSuggestion(...)` | 生成修复建议（供漏洞管理使用） |
| `summaryReport(...)` | 安全报告总结（供全系统使用） |

#### 4.3.3 全局AI调用预留位置
在**主动漏洞扫描流程**中预留标准AI调用点（注释形式），未来接入AI后统一启用：
1. 扫描前：// 【AI全局调用】生成智能Payload
2. 研判时：// 【AI全局调用】AI智能验证漏洞
3. 评级时：// 【AI全局调用】风险等级智能评估
4. 入库前：// 【AI全局调用】自动生成修复建议
5. 完成后：// 【AI全局调用】智能生成扫描总结

### 4.4 扩展说明（全局AI能力）
本阶段仅完成AI配置与调用入口预留，**不实现具体AI逻辑**。
未来可基于该全局AI服务，逐步扩展至：
- 流量智能分析与异常检测
- 攻击类型自动识别与归类
- 安全告警自动研判与降噪
- 防御策略智能推荐与生成
  真正实现**AI驱动的一体化智能网络监测系统**。

## 五、漏洞新增/编辑功能设计（重点解决手动操作数据库问题）
结合现有系统现状（目前仅能通过数据库操作新增/修改漏洞），建议**在现有漏洞监测页面扩展新增/编辑功能**，不单独新增页面，既保证操作便捷性，又保持页面结构简洁，贴合毕设实用性要求，具体设计如下：

### 5.1 功能定位
实现漏洞的前端手动新增、编辑功能，替代直接操作数据库的方式，与自动扫描漏洞、未来AI扫描漏洞形成“手动+自动”双模式漏洞录入体系，统一在漏洞监测页面管理，降低操作成本，提升系统完整性。

### 5.2 前端扩展（基于现有漏洞监测页面）
#### 5.2.1 页面扩展
在现有漏洞监测页面的筛选栏下方、表格上方，新增2个操作按钮：
- 【新增漏洞】：点击弹出新增漏洞弹窗
- 【编辑漏洞】：表格操作列新增“编辑”按钮，点击弹出编辑弹窗（与详情弹窗复用部分组件，简化开发）

#### 5.2.2 新增/编辑弹窗设计
弹窗包含以下表单字段（与VulnerabilityMonitorEntity实体类字段对应，复用现有详情弹窗样式）：
| 字段 | 类型 | 说明 | 必选 |
|------|------|------|------|
| 漏洞名称 | 文本输入 | 漏洞的具体名称（如“SQL注入漏洞 - 登录绕过”） | 是 |
| 漏洞类型 | 下拉选择 | 可选值：SQL_INJECTION、XSS、COMMAND_INJECTION等 | 是 |
| 风险等级 | 下拉选择 | 可选值：CRITICAL、HIGH、MEDIUM、LOW | 是 |
| 漏洞路径 | 文本输入 | 漏洞所在接口路径（如“/api/login”） | 是 |
| 验证状态 | 下拉选择 | 可选值：未验证（0）、已验证可利用（1） | 是 |
| 漏洞描述 | 文本域 | 漏洞的详细描述（如“登录接口存在SQL注入，可绕过身份验证”） | 否 |
| 修复建议 | 文本域 | 漏洞的修复方案（如“使用参数化查询，过滤用户输入”） | 否 |

#### 5.2.3 JS交互扩展（基于现有vulnerability-monitor.js）
新增4个核心交互方法，不影响现有逻辑：
1. `openAddVulnModal()`：打开新增漏洞弹窗，重置表单数据。
2. `openEditVulnModal(id)`：打开编辑漏洞弹窗，根据漏洞ID查询详情并回显表单。
3. `saveVuln()`：校验表单数据，调用后端新增/编辑接口，保存漏洞信息。
4. `deleteVuln(id)`：表格操作列新增“删除”按钮，调用后端删除接口，删除漏洞（可选，提升完整性）。

### 5.3 后端扩展（基于现有代码）
#### 5.3.1 控制器层扩展（VulnerabilityMonitorController）
在现有接口基础上，新增3个接口，用于新增、编辑、删除漏洞：
| 接口 | 方法 | 功能 |
|------|------|------|
| `/api/vulnerability/add` | POST | 新增手动预设漏洞 |
| `/api/vulnerability/edit` | POST | 编辑现有漏洞（手动/自动扫描均可） |
| `/api/vulnerability/delete/{id}` | DELETE | 删除漏洞（可选，建议保留，便于管理） |

#### 5.3.2 服务层扩展（VulnerabilityVerifyServiceImpl）
新增对应方法，调用现有Mapper接口实现数据操作，确保与现有缓存、统计逻辑兼容：
1. `addVulnerability(VulnerabilityMonitorEntity vuln)`：新增漏洞，同步更新内存缓存。
2. `editVulnerability(VulnerabilityMonitorEntity vuln)`：编辑漏洞，同步更新内存缓存。
3. `deleteVulnerability(Long id)`：删除漏洞，同步删除内存缓存中的对应数据。

#### 5.3.3 数据层兼容
无需修改现有实体类、Mapper接口，新增/编辑的漏洞数据直接写入sys_vulnerability_monitor表，与自动扫描、未来AI扫描的漏洞数据统一存储，确保现有筛选、统计、联动逻辑正常生效。

### 5.4 设计优势
1. 操作便捷：在现有漏洞监测页面直接扩展，无需切换页面，管理员可快速完成漏洞新增/编辑。
2. 结构简洁：复用现有页面样式、弹窗组件，减少开发量，保持系统界面统一性。
3. 功能完整：替代数据库手动操作，提升系统实用性，符合毕设“可落地、可操作”的要求。
4. 兼容扩展：新增/编辑的漏洞与自动扫描、AI扫描的漏洞统一管理，不影响现有功能。

## 六、整体页面结构（最终版）
```
漏洞管理

├─ 漏洞扫描（新增）
│   ├─ 扫描配置：扫描目标、扫描类型

└─ 系统配置（已完成）
    ├─ 系统通用配置
    └─ 全局AI大模型配置（url、apiKey）
```

## 七、核心设计亮点
1. 兼容性：完全保留现有功能，新增功能采用扩展式开发，不修改原有代码，降低开发风险。
2. 双模式管理：支持手动预设漏洞与自动扫描漏洞，未来可扩展AI扫描，满足不同场景需求。
3. 全局可扩展性：AI设计为**系统全局能力**，一次配置全模块复用，支持未来扩展流量分析、攻击识别、智能告警等功能。
4. 实用性：新增前端漏洞新增/编辑功能，替代数据库手动操作，提升系统可操作性，贴合毕设落地要求。
5. 闭环联动：主动扫描发现漏洞→自动入库→漏洞监测页面管理→攻击联动统计→防御决策，形成完整安全监测闭环。

## 八、开发优先级建议
1. 第一优先级：扩展现有漏洞监测页面，实现漏洞新增/编辑/删除功能（解决手动操作数据库的痛点）。
2. 第二优先级：开发漏洞扫描页面及后端扫描服务，实现基础主动扫描功能。
3. 第三优先级：在现有系统配置页面补充AI配置项，完成全局AiModelService预留。
4. 第四优先级（后续）：接入第三方AI大模型，补全全局AI能力，逐步扩展至全监测系统。
