## 二、状态机逻辑优化方案

### 2.1 当前问题分析

当前状态机存在以下问题：

1. **状态转换太快** - 缺少观察期
2. **缺少周期推送** - 只有状态变化时才推送
3. **请求内容未区分** - 不同接口的攻击被当作同一处理
4. **状态独立性** - 每个IP的状态应该是独立的

### 2.2 优化后的状态机设计

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        IP状态机（每个IP独立）                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ┌──────────┐                                                              │
│   │  NORMAL  │ ← 正常请求，正常记录推送                                       │
│   └────┬─────┘                                                              │
│        │                                                                    │
│        │ 短时间内请求频率超过阈值（如10次/秒）                                  │
│        │ 触发限流 → 记录限流日志 + 流量数据                                    │
│        ▼                                                                    │
│   ┌──────────┐                                                              │
│   │SUSPICIOUS│ ← 可疑状态，重点关注该IP                                       │
│   └────┬─────┘   • 按周期推送统计数据（如每3秒）                               │
│        │        • 区分不同请求内容，分别统计                                   │
│        │        • 继续执行限流                                                │
│        │                                                                    │
│        │ 持续高频请求（连续限流N次）                                           │
│        │ 或攻击不同接口                                                       │
│        ▼                                                                    │
│   ┌──────────┐                                                              │
│   │ ATTACKING│ ← 攻击中，准备防御                                            │
│   └────┬─────┘   • 记录攻击事件                                              │
│        │        • 按周期推送统计数据                                          │
│        │        • 区分不同请求内容                                            │
│        │                                                                    │
│        │ 达到防御阈值                                                         │
│        ▼                                                                    │
│   ┌──────────┐                                                              │
│   │ DEFENDED │ ← 已防御，执行封禁                                            │
│   └────┬─────┘   • 采样推送（区分不同请求内容）                               │
│        │        • 推送黑名单事件                                              │
│        │        • 拦截请求但记录流量                                          │
│        │                                                                    │
│        │ 攻击停止（一段时间无请求）                                            │
│        ▼                                                                    │
│   ┌──────────┐                                                              │
│   │ COOLDOWN │ ← 冷却期，观察是否恢复                                         │
│   └────┬─────┘   • 按周期推送统计数据                                         │
│        │        • 正常放行请求                                               │
│        │                                                                    │
│        │ 冷却期结束，无异常行为                                               │
│        │                                                                    │
│        └──────────────→ NORMAL                                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.3 状态转换详细说明

| 当前状态 | 触发条件 | 目标状态 | 数据处理 |
|----------|----------|----------|----------|
| NORMAL | 请求频率超过阈值，触发限流 | SUSPICIOUS | 推送状态变化前的统计数据 |
| SUSPICIOUS | 持续高频请求（连续限流N次） | ATTACKING | 推送周期数据 |
| SUSPICIOUS | 一段时间无异常请求 | NORMAL | 推送状态变化前的统计数据 |
| ATTACKING | 达到防御阈值 | DEFENDED | 推送攻击事件，执行封禁 |
| DEFENDED | 攻击停止（一段时间无请求） | COOLDOWN | 推送状态变化 |
| COOLDOWN | 冷却期结束，无异常 | NORMAL | 推送状态变化 |

### 4.2 聚合Key生成规则

```java
private String buildAggregateKey(TrafficMonitorDTO traffic) {
    StringBuilder keyBuilder = new StringBuilder();
    keyBuilder.append(nullToEmpty(traffic.getSourceIp())).append("|");
    keyBuilder.append(nullToEmpty(traffic.getTargetIp())).append("|");
    keyBuilder.append(nullToEmpty(traffic.getRequestUri())).append("|");
    keyBuilder.append(nullToEmpty(traffic.getHttpMethod())).append("|");
    keyBuilder.append(nullToEmpty(traffic.getContentType())).append("|");
    keyBuilder.append(normalizeHeaders(traffic.getRequestHeaders())).append("|");
    keyBuilder.append(normalizeBody(traffic.getRequestBody()));
    
    return DigestUtils.md5DigestAsHex(keyBuilder.getBytes(StandardCharsets.UTF_8));
}
```

## 五、目标IP不统一问题

### 5.1 问题分析

查看流量记录，目标IP出现多种值：
- `0.0.0.0`
- `localhost`

**原因**：`ServerWebExchangeUtil.extractTargetIp()` 方法提取逻辑不完善

### 5.2 解决方案

目标IP提取优先级：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        目标IP提取逻辑                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   1. Host请求头                                                              │
│      - 提取主机名/IP部分（去除端口）                                          │
│      - localhost → 127.0.0.1                                                │
│      - 域名保持原样                                                          │
│                                                                             │
│   2. 请求本地地址                                                            │
│      - 规范化IPv6地址                                                        │
│      - 转换为标准格式                                                        │
│                                                                             │
│   3. 默认值                                                                  │
│      - 统一为 0.0.0.0                                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 5.3 修复代码

```java
public static String extractTargetIp(ServerHttpRequest request) {
    String host = request.getHeaders().getFirst(HttpHeaders.HOST);
    
    if (StringUtils.hasText(host)) {
        if (host.contains(":")) {
            host = host.split(":")[0].trim();
        }
        if ("localhost".equalsIgnoreCase(host)) {
            return "127.0.0.1";
        }
        if (isValidIpAddress(host)) {
            return IpNormalizeUtil.normalize(host);
        }
        return host.trim();
    }
    
    InetSocketAddress localAddress = request.getLocalAddress();
    if (localAddress != null) {
        String localIp = localAddress.getAddress().getHostAddress();
        return IpNormalizeUtil.normalize(localIp);
    }
    
    return "0.0.0.0";
}
```

## 七、黑名单首次标识问题

### 7.1 问题分析

同一IP第二次被封禁时仍显示"BLACKLIST首次"

**原因**：
- 黑名单事件推送时未检查历史记录
- `defenseType` 字段固定为 "BLACKLIST首次"

### 7.2 解决方案

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        黑名单首次标识逻辑                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   推送黑名单事件前：                                                          │
│                                                                             │
│   1. 查询该IP的历史封禁记录                                                   │
│      SELECT COUNT(*) FROM sys_ip_blacklist                                  │
│      WHERE ip = ? AND create_time < ?                                       │
│                                                                             │
│   2. 根据查询结果设置标识                                                     │
│      - 有历史记录 → defenseType = "BLACKLIST重复"                            │
│      - 无历史记录 → defenseType = "BLACKLIST首次"                            │
│                                                                             │
│   3. 更新黑名单表                                                            │
│      - is_repeat_offender = true/false                                      │
│      - offense_count = 历史次数 + 1                                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

4. **周期推送机制**
   - 每3秒检查一次需要推送的数据
   - 按请求内容分组统计
   - 支持采样保存

5. **状态机优化**
   - 每个IP状态独立管理
   - 支持聚合数据收集
   - 周期推送与状态变化推送结合
