## 二、问题详细分析

### 2.1 P1: 告警API路径重复

**问题现象**：
```
GET http://localhost:9002/api/api/alert/stats 404 (Not Found)
```

**根本原因**：

| 文件 | 定义 | 问题 |
|------|------|------|
| `config.js` | `API_BASE_URL: '/api'` | 基础路径已包含 `/api` |
| `http.js` | `fullUrl = baseUrl + url` | 自动拼接基础路径 |
| `alert-manage.js` | `http.get('/api/alert/stats')` | 调用时又加了 `/api` |

**最终URL**：`/api` + `/api/alert/stats` = `/api/api/alert/stats` ❌

**解决方案**：
- 修改 `alert-manage.js`，将所有 `/api/xxx` 改为 `/xxx`

### 2.2 P2: 限流计数逻辑错误

**问题现象**：
- 没有限流过程，直接触发黑名单
- 状态转换太快

**当前代码** ([RequestRateLimitFilter.java:89](file:///c:/Users/1/Documents/AAAABS/network-monitor-parent/gateway-service/src/main/java/com/network/gateway/filter/defense/RequestRateLimitFilter.java#L89))：
```java
// 问题：每次请求都增加计数
int currentRateLimitCount = attackStateCache.incrementRateLimitCount(sourceIp, windowMs);
```

**问题分析**：
- `incrementRateLimitCount` 在每次请求时都被调用
- 应该只在限流触发时才增加计数

**解决方案**：
- 只有当 `isRateLimited = true` 时才增加限流计数

### 2.3 P3: 被拦截请求未记录流量数据

**问题现象**：
- DEFENDED状态的请求直接拦截，没有流量记录
- 分析数据不完整，无法准确判断攻击态势

**当前流程**：
```
请求 → 状态检查 → [DEFENDED?] → 直接拦截（无记录）
                    ↓
               限流检查 → [超限?] → 拦截（只记防御日志）
                    ↓
               放行 → 记录流量
```

**问题**：
- 被拦截的请求也是攻击行为的一部分
- 缺少这些数据会导致分析结果不准确

**正确理念**：
> **数据采集先于防御判断** - 所有请求都先记录，防御拦截不影响数据完整性

**解决方案**：
```
请求 → 数据采集（完整记录）→ 状态检查 → [DEFENDED?] → 拦截响应
                                      ↓
                                 限流检查 → [超限?] → 拦截响应
                                      ↓
                                 放行
```

### 2.4 P4: 状态转换阈值过低

**当前配置** ([IpAttackStateConstant.java](file:///c:/Users/1/Documents/AAAABS/network-monitor-parent/gateway-service/src/main/java/com/network/gateway/constant/IpAttackStateConstant.java))：

| 常量 | 当前值 | 说明 |
|------|--------|------|
| `SUSPICIOUS_RATE_LIMIT_THRESHOLD` | 2 | 2次请求就变可疑 |
| `ATTACKING_RATE_LIMIT_THRESHOLD` | 5 | 5次请求就变攻击中 |
| `ATTACKING_DURATION_MS` | 30000 | 攻击状态持续30秒后才防御 |

**问题**：
- 阈值过低，正常请求也可能触发
- ATTACKING → DEFENDED 需要等待30秒，不够及时

**解决方案**：

| 项目 | 当前值 | 优化值 | 说明 |
|------|--------|--------|------|
| SUSPICIOUS阈值 | 2次请求 | 3次限流触发 | 连续限流3次才变可疑 |
| ATTACKING阈值 | 5次请求 | 5次限流触发 | 连续限流5次才变攻击中 |
| DEFENDED触发 | 等待30秒 | 达到阈值立即执行 | 更及时的防御响应 |

### 2.5 P5: 缺少限流阶段的防御日志

**问题现象**：
- 防御日志只有黑名单记录
- 没有限流拦截的日志

**原因**：
- 状态直接从 NORMAL → SUSPICIOUS → ATTACKING → DEFENDED
- 没有经过 `handleRateLimitExceeded` 方法

**解决方案**：
- 确保限流拦截时记录防御日志
- 每个状态转换都记录日志

### 2.6 P6: 攻击记录不完整

**问题现象**：
- 攻击监测只有1条记录
- 没有完整记录攻击链

**原因**：
- 事件创建后，后续请求被拦截
- 没有新记录产生

**解决方案**：
- 每个状态转换都创建对应记录
- 限流阶段也创建攻击记录
- 完整记录攻击链：发现 → 可疑 → 攻击 → 防御 → 冷却

### 2.7 P7: 聚合Key过于简单

**问题现象**：
- 不同请求被错误聚合到一起
- 导致数据不完整

**当前代码** ([AggregatePushHandler.java:111](file:///c:/Users/1/Documents/AAAABS/network-monitor-parent/gateway-service/src/main/java/com/network/gateway/handler/AggregatePushHandler.java#L111))：
```java
private String buildAggregateKey(TrafficMonitorDTO traffic) {
    return traffic.getSourceIp();  // 只用IP作为Key
}
```

**问题**：
- 只有IP相同就聚合
- 不同的URL、请求体、请求头都被聚合在一起
- 导致数据丢失

**正确的聚合逻辑**：

> **只有请求完全相同才能聚合**
> - 请求IP相同
> - 请求URL相同
> - 请求方法相同
> - 请求头相同
> - 请求体相同
> - 目标端口相同
> - 请求时间在统计时间范围内（不要求完全相同）

**解决方案**：
```java
private String buildAggregateKey(TrafficMonitorDTO traffic) {
    StringBuilder keyBuilder = new StringBuilder();
    keyBuilder.append(traffic.getSourceIp()).append("|");
    keyBuilder.append(traffic.getTargetIp()).append("|");
    keyBuilder.append(traffic.getRequestUri()).append("|");
    keyBuilder.append(traffic.getHttpMethod()).append("|");
    keyBuilder.append(normalizeHeaders(traffic.getRequestHeaders())).append("|");
    keyBuilder.append(normalizeBody(traffic.getRequestBody())).append("|");
    keyBuilder.append(traffic.getContentType());
    return DigestUtils.md5Hex(keyBuilder.toString());
}
```

### 4.4 优化聚合Key生成逻辑

**文件**：`gateway-service/src/main/java/com/network/gateway/handler/AggregatePushHandler.java`

**修改内容**：
```java
private String buildAggregateKey(TrafficMonitorDTO traffic) {
    StringBuilder keyBuilder = new StringBuilder();
    keyBuilder.append(nullToEmpty(traffic.getSourceIp())).append("|");
    keyBuilder.append(nullToEmpty(traffic.getTargetIp())).append("|");
    keyBuilder.append(nullToEmpty(traffic.getRequestUri())).append("|");
    keyBuilder.append(nullToEmpty(traffic.getHttpMethod())).append("|");
    keyBuilder.append(nullToEmpty(traffic.getContentType())).append("|");
    keyBuilder.append(normalizeHeaders(traffic.getRequestHeaders())).append("|");
    keyBuilder.append(normalizeBody(traffic.getRequestBody()));
    return DigestUtils.md5Hex(keyBuilder.toString());
}

private String normalizeHeaders(String headers) {
    if (headers == null || headers.isEmpty()) {
        return "";
    }
    return headers.lines()
        .sorted()
        .collect(Collectors.joining(";"));
}

private String normalizeBody(String body) {
    if (body == null || body.isEmpty()) {
        return "";
    }
    if (body.length() > 500) {
        return DigestUtils.md5Hex(body);
    }
    return body;
}

private String nullToEmpty(String str) {
    return str == null ? "" : str;
}
```
