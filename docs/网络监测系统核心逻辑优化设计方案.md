## 二、网关服务唤醒机制设计（核心优化）

### 2.1 设计原则

网关服务采用**事件驱动唤醒机制**，摒弃传统的循环轮询模式，实现以下核心原则：

1. **按需唤醒**：只有当有流量进入时才唤醒相关处理逻辑，无流量时完全休眠
2. **零空转**：不存在后台定时循环刷新等无效操作，避免CPU资源浪费
3. **即时响应**：请求到达时立即触发处理流程，无延迟
4. **资源高效**：无流量时CPU占用率趋近于零

### 2.3 与传统循环模式对比

| 对比项 | 传统循环模式 | 事件驱动唤醒模式 |
|--------|--------------|------------------|
| CPU占用（无流量） | 持续占用（定时器、轮询） | 接近零 |
| 内存占用 | 常驻对象、定时任务队列 | 仅事件处理时分配 |
| 响应延迟 | 定时器间隔内延迟 | 即时响应 |

| 扩展性 | 定时器数量随功能增加 | 无额外开销 |
| 资源利用率 | 低效，空转浪费 | 高效，按需使用 |

### 2.4 具体实现机制

#### 2.4.1 流量采集唤醒

```java
/**
 * 流量采集过滤器 - 事件驱动模式
 * 只有请求到达时才执行采集逻辑，无请求时完全休眠
 */
public class TrafficCollectGlobalFilter implements GlobalFilter, Ordered {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 请求到达，立即唤醒采集逻辑
        long startTime = System.currentTimeMillis();
        String ip = getClientIp(exchange);
        
        return chain.filter(exchange)
            .doOnTerminate(() -> {
                // 请求处理完成，触发流量记录
                long processingTime = System.currentTimeMillis() - startTime;
                int statusCode = exchange.getResponse().getStatusCode().value();
                
                // 唤醒流量队列管理器，记录本次请求
                trafficQueueManager.onRequestArrived(ip, exchange, statusCode, processingTime);
                
                // 检查是否满足推送条件（有新数据才检查，无数据不检查）
                trafficQueueManager.checkAndPushIfNeeded(ip);
            });
    }
}
```

#### 2.4.2 推送队列唤醒

```java
/**
 * 推送队列管理器 - 按需推送
 * 不使用定时器，而是在每次请求到达时检查是否需要推送
 */
public class TrafficQueueManager {
    
    /**
     * 请求到达时调用，记录流量数据
     */
    public void onRequestArrived(String ip, ServerWebExchange exchange, int statusCode, long processingTime) {
        IpTrafficQueue queue = getOrCreateQueue(ip);
        queue.addRequest(exchange, statusCode, processingTime);
        
        // 唤醒状态机检查（有请求才检查）
        stateMachineService.onRequestArrived(ip);
    }
    
    /**
     * 检查是否需要推送（有新数据才检查）
     */
    public void checkAndPushIfNeeded(String ip) {
        IpTrafficQueue queue = getQueue(ip);
        if (queue == null || queue.isEmpty()) {
            return; // 无数据，不推送
        }
        
        // 检查推送条件（满足任一即推送）
        if (shouldPushByInterval(queue) ||      // 周期性推送
            shouldPushByQueueFull(queue) ||     // 队列满
            shouldPushByStateChange(queue)) {   // 状态切换
            pushToMonitorService(queue);
        }
    }
    
    /**
     * 周期性推送检查（基于上次推送时间和当前请求时间）
     */
    private boolean shouldPushByInterval(IpTrafficQueue queue) {
        long now = System.currentTimeMillis();
        long lastPushTime = queue.getLastPushTime();
        long interval = configCache.getLong("traffic.push.interval-ms", 3000L);
        
        // 只有当距离上次推送超过间隔，且有新数据时才推送
        return (now - lastPushTime) >= interval && queue.hasNewData();
    }
}
```

#### 2.4.3 状态机唤醒

```java
/**
 * 状态机服务 - 事件驱动
 * 只有请求到达时才检查状态转换，无请求时不检查
 */
public class StateMachineService {
    
    /**
     * 请求到达时调用，检查状态转换
     */
    public void onRequestArrived(String ip) {
        IpAttackStateEntry entry = stateCache.getEntry(ip);
        if (entry == null) {
            return;
        }
        
        // 更新请求计数
        entry.incrementRequestCount();
        
        // 检查状态转换条件（有请求才检查）
        checkStateTransition(entry);
    }
    
    /**
     * 检查状态转换（基于当前状态和请求行为）
     */
    private void checkStateTransition(IpAttackStateEntry entry) {
        int currentState = entry.getState();
        
        switch (currentState) {
            case NORMAL:
                checkNormalToSuspicious(entry);
                break;
            case SUSPICIOUS:
                checkSuspiciousToAttacking(entry);
                checkSuspiciousToNormal(entry);
                break;
            case ATTACKING:
                checkAttackingToDefended(entry);
                break;
            case DEFENDED:
                checkDefendedToCooldown(entry);
                break;
            case COOLDOWN:
                checkCooldownToNormal(entry);
                checkCooldownToAttacking(entry);
                break;
        }
    }
}
```

### 2.5 性能优势

1. **CPU利用率**：无流量时CPU占用率从传统模式的5-10%降低到接近0%
2. **内存占用**：无定时器队列、无后台线程，内存占用减少30-50%
3. **响应延迟**：消除定时器间隔延迟，响应时间缩短至毫秒级
4. **扩展性**：支持更高并发，无定时器数量限制

## 三、配置统一管理与同步机制

### 3.1 设计原则

整个系统的配置统一由监测服务（monitor-service）的系统配置页面进行管理，实现以下核心原则：

1. **单一数据源**：所有配置存储在数据库 `sys_config` 表中，由监测服务统一管理
2. **无硬编码**：网关服务启动时从监测服务同步配置，不进行硬编码
3. **动态同步**：配置变更时自动同步到网关服务，无需重启
4. **一致性保证**：采用类TCP三次握手机制确保配置同步状态一致

### 3.4 配置同步接口设计

#### 3.4.1 监测服务端接口

```java
/**
 * 配置同步接口（监测服务端）
 */
@RestController
@RequestMapping("/api/inner/config")
public class ConfigSyncController {
    
    /**
     * 配置同步接口（第一次握手）
     * 网关请求同步配置
     */
    @PostMapping("/sync")
    public ApiResponse<ConfigSyncResponse> syncConfig(@RequestBody ConfigSyncRequest request) {
        // 获取当前配置版本
        String currentVersion = configService.getConfigVersion();
        
        // 如果网关版本与当前版本一致，返回无需更新
        if (request.getVersion() != null && request.getVersion().equals(currentVersion)) {
            return ApiResponse.success(ConfigSyncResponse.noUpdate(currentVersion));
        }
        
        // 返回最新配置
        Map<String, String> configs = configService.getAllConfigs();
        return ApiResponse.success(ConfigSyncResponse.update(currentVersion, configs));
    }
    
    /**
     * 配置确认接口（第三次握手）
     * 网关确认已收到配置
     */
    @PostMapping("/ack")
    public ApiResponse<Void> ackConfig(@RequestBody ConfigAckRequest request) {
        // 记录网关同步状态
        configService.recordSyncStatus(request.getService(), request.getVersion(), request.getStatus());
        
        // 记录同步日志
        log.info("配置同步确认: service={}, version={}, status={}", 
            request.getService(), request.getVersion(), request.getStatus());
        
        return ApiResponse.success();
    }
    
    /**
     * 配置变更通知接口（被动触发）
     * 管理员修改配置后，通知网关重新同步
     */
    @PostMapping("/notify-change")
    public ApiResponse<Void> notifyConfigChange(@RequestBody ConfigChangeNotifyRequest request) {
        // 通知所有网关实例重新同步
```

```java
/**
 * 配置同步客户端（网关服务端）
 */
@Component
public class ConfigSyncClient {
    
    /**
     * 服务启动时同步配置
     */
    @PostConstruct
    public void syncConfigOnStartup() {
        // 执行三次握手同步
        boolean success = doThreeWayHandshake();
        
        if (!success) {
            // 同步失败，使用默认配置并记录告警
            log.warn("配置同步失败，使用默认配置");
            useDefaultConfig();
        }
    }
    
    /**
     * 执行三次握手同步
     */
    private boolean doThreeWayHandshake() {
        int maxRetry = 3;
        
        for (int i = 0; i < maxRetry; i++) {
            try {
                // 第一次握手：请求同步
                ConfigSyncResponse response = requestSync();
                
                if (!response.needUpdate()) {
                    return true; // 无需更新
                }
                
                // 更新本地配置缓存
                updateLocalConfig(response.getConfigs());
                
                // 第三次握手：确认收到
                ackSync(response.getVersion());
                
                log.info("配置同步成功: version={}", response.getVersion());
                return true;
                
            } catch (Exception e) {
                log.warn("配置同步失败，尝试重试 ({}/{})", i + 1, maxRetry, e);
                sleep(1000 * (i + 1)); // 指数退避
            }
        }
        
        return false;
    }
    
    /**
     * 接收配置变更通知（被动触发）
     */
    @PostMapping("/api/gateway/config/notify")
    public ApiResponse<Void> onConfigNotify(@RequestBody ConfigChangeNotifyRequest request) {
        log.info("收到配置变更通知: newVersion={}", request.getNewVersion());
        
        // 触发重新同步
        doThreeWayHandshake();
        
        return ApiResponse.success();
    }
}
```

### 3.6 配置同步流程说明

1. **启动同步**：网关服务启动时，主动向监测服务请求配置同步
2. **三次握手**：
   - 第一次握手：网关发送同步请求（SYN），携带当前配置版本
   - 第二次握手：监测服务返回最新配置（SYN-ACK），如版本一致则无需更新
   - 第三次握手：网关确认收到配置（ACK），完成同步
3. **连接断开**：握手完成后，HTTP连接断开，无需保持持久连接
4. **变更通知**：管理员修改配置后，监测服务主动通知网关重新同步
5. **重新同步**：网关收到通知后，再次执行三次握手同步新配置

### 3.7 配置同步优势

| 优势 | 说明 |
|------|------|
| 无硬编码 | 所有配置从数据库读取，网关服务无硬编码配置 |
| 动态更新 | 配置变更后自动同步，无需重启服务 |
| 一致性保证 | 三次握手机制确保配置同步状态一致 |
| 资源高效 | 无需保持持久连接，仅在需要时同步 |
| 可追溯 | 记录配置版本和同步状态，便于审计和排查 |

### 4.5 状态机与置信度的联动关系（核心补充）

1. **状态机变化不直接依赖置信度计算结果**
    状态机流转**核心驱动力为IP实时行为**（请求频率、持续异常时长、URI访问多样性、静默恢复时长等规则），行为规则独立生效并驱动状态切换，置信度仅作为**风险评分辅助决策**，不独立驱动状态机流转。

2. **置信度的作用边界**
    仅用于评估攻击风险可信度、决定是否从`ATTACKING`升级至`DEFENDED`、区分误判与真实攻击、调整防御强度，不控制状态机基础流转逻辑。

3. **联动执行流程**
    ```
    IP请求行为发生 → 行为规则触发状态机切换 → 新状态下计算置信度 → 置信度达标触发梯度防御 → 防御动作推动状态机最终确认
    ```

## 五、流量采集与推送重构方案

### 5.1 核心设计原则（补充优化原则）

1.  **数据真实性**：保留真实请求信息，不伪造数据，杜绝`COUNTER/__counter__`篡改；
2.  **时序正确性**：使用队列机制保证推送顺序，新增失败重试与降级策略；
3.  **聚合合理性**：按 IP + 状态 + URI模式 分组聚合，降低推送压力；
4.  **样本保留**：保留关键样本用于分析，新增样本淘汰与脱敏规则；
5.  **性能优化**：轻量化采集、高效匹配，适配高并发场景；
6.  **容错性**：推送失败有重试、队列溢出有降级，避免数据丢失；
7.  **事件驱动**：采用事件驱动模式，无流量时完全休眠，有流量时才唤醒处理。

### 6.1 置信度计算公式（优化完善）

```
confidence = baseScore 
           + frequencyScore    // 频率异常程度
           + diversityScore    // 攻击多样性
           + persistenceScore  // 持续时间
           + patternScore      // 攻击模式匹配
           + historyScore      // 历史行为权重（新增）
```

## 七、防御后流量处理方案（新增核心优化）

### 7.1 问题描述

当防御措施执行后（如限流、黑名单封禁），存在以下严重问题：

1. **响应状态显示错误**：被拦截的请求在流量监测列表中仍显示响应状态200，实际应显示429（限流）或403（黑名单拦截）
2. **被拦截流量未纳入判定**：被拦截的流量未参与置信度计算和状态判定，导致：
   - 持续攻击时置信度不再变化
   - 状态机在DEFENDED状态后不再响应持续攻击行为
3. **安全漏洞**：攻击者可利用此漏洞，触发防御后继续攻击，由于被拦截流量不计入判定，系统可能误判攻击已停止

### 7.2 解决方案

#### 7.2.1 响应状态正确记录

```java
/**
 * 流量采集过滤器 - 防御后状态记录优化
 */
public class TrafficCollectGlobalFilter implements GlobalFilter, Ordered {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // ... 前置处理 ...
        
        return chain.filter(exchange)
            .doOnSuccess(aVoid -> {
                // 正常响应，记录实际状态码
                int statusCode = exchange.getResponse().getStatusCode().value();
                recordTraffic(exchange, statusCode);
            })
            .doOnError(throwable -> {
                // 错误响应
                recordTraffic(exchange, 500);
            })
            .doOnTerminate(() -> {
                // 新增：检查是否被防御拦截
                checkDefenseIntercept(exchange);
            });
    }
    
    /**
     * 检查防御拦截状态
     */
    private void checkDefenseIntercept(ServerWebExchange exchange) {
        String ip = getClientIp(exchange);
        
        // 检查是否被限流
        Boolean rateLimited = exchange.getAttribute("rate_limited");
        if (Boolean.TRUE.equals(rateLimited)) {
            // 更新流量记录为429状态
            updateTrafficStatus(exchange, 429);
            // 新增：将被拦截请求纳入判定
            handleBlockedTraffic(ip, "RATE_LIMIT", 429);
        }
        
        // 检查是否被黑名单拦截
        Boolean blacklisted = exchange.getAttribute("blacklisted");
        if (Boolean.TRUE.equals(blacklisted)) {
            // 更新流量记录为403状态
            updateTrafficStatus(exchange, 403);
            // 新增：将被拦截请求纳入判定
            handleBlockedTraffic(ip, "BLACKLIST", 403);
        }
    }
    
    /**
     * 处理被拦截的流量（核心新增）
     * 将被拦截的流量纳入置信度计算和状态判定
     */
    private void handleBlockedTraffic(String ip, String blockType, int statusCode) {
        // 1. 更新IP流量队列的拦截计数
        IpTrafficQueue queue = trafficQueueManager.getOrCreateQueue(ip);
        queue.incrementBlockedCount(blockType);
        
        // 2. 触发置信度计算（被拦截请求也应计入攻击判定）
        confidenceService.calculateOnBlocked(ip, blockType);
        
        // 3. 更新状态机（持续攻击应维持或升级防御状态）
        stateMachineService.onBlockedRequest(ip, blockType);
        
        // 4. 记录防御日志（去重合并）
        defenseLogService.recordBlock(ip, blockType, statusCode);
    }
}
```

#### 7.2.2 被拦截流量纳入置信度计算

```java
/**
 * 置信度服务 - 防御后计算优化
 */
public class ConfidenceService {
    
    /**
     * 被拦截请求触发置信度计算（新增）
     */
    public void calculateOnBlocked(String ip, String blockType) {
        IpTrafficQueue queue = trafficQueueManager.getQueue(ip);
        if (queue == null) return;
        
        // 获取当前置信度
        int currentConfidence = queue.getConfidence();
        
        // 被拦截请求增加置信度（表明攻击仍在持续）
        int blockedScore = getBlockedScore(blockType);
        int newConfidence = Math.min(100, currentConfidence + blockedScore);
        
        // 更新置信度
        queue.setConfidence(newConfidence);
        
        // 记录置信度变化
        logConfidenceChange(ip, currentConfidence, newConfidence, "BLOCKED:" + blockType);
    }
    
    /**
     * 获取被拦截请求的得分（新增）
     */
    private int getBlockedScore(String blockType) {
        switch (blockType) {
            case "RATE_LIMIT":
                return configCache.getInt("confidence.blocked.rate-limit-score", 3);
            case "BLACKLIST":
                return configCache.getInt("confidence.blocked.blacklist-score", 5);
            default:
                return 2;
        }
    }
}
```

#### 7.2.3 被拦截流量纳入状态判定

```java
/**
 * 状态机服务 - 防御后状态判定优化
 */
public class StateMachineService {
    
    /**
     * 被拦截请求触发状态判定（新增）
     */
    public void onBlockedRequest(String ip, String blockType) {
        IpAttackStateCache.StateInfo stateInfo = stateCache.getStateInfo(ip);
        if (stateInfo == null) return;
        
        int currentState = stateInfo.getState();
        
        // 根据当前状态和拦截类型进行状态判定
        switch (currentState) {
            case IpState.NORMAL:
            case IpState.SUSPICIOUS:
                // 正常/可疑状态被拦截，说明攻击确认，升级状态
                transitionTo(ip, IpState.ATTACKING, "连续被拦截确认攻击");
                break;
                
            case IpState.ATTACKING:
                // 攻击中状态被拦截，升级为已防御
                transitionTo(ip, IpState.DEFENDED, "防御措施生效");
                break;
                
            case IpState.DEFENDED:
                // 已防御状态仍被拦截，说明攻击持续
                // 更新拦截计数，延长封禁时间
                updateDefenseMetrics(ip, blockType);
                // 重置冷却期计时器（攻击仍在持续）
                resetCooldownTimer(ip);
                break;
                
            case IpState.COOLDOWN:
                // 冷却期被拦截，重新进入攻击状态
                transitionTo(ip, IpState.ATTACKING, "冷却期内再次攻击");
                break;
        }
    }
    
    /**
     * 更新防御指标（新增）
     */
    private void updateDefenseMetrics(String ip, String blockType) {
        // 增加拦截计数
        stateCache.incrementBlockedCount(ip);
        
        // 获取累计拦截次数
        int blockedCount = stateCache.getBlockedCount(ip);
        
        // 根据拦截次数动态延长封禁时间
        if (blockedCount >= configCache.getInt("defense.extend-threshold", 10)) {
            extendBanDuration(ip, blockedCount);
        }
    }
}
```

## 十三、总结

本优化方案从以下几个核心维度对网络监测系统进行了全面优化：

1. **网关服务唤醒机制**：摒弃传统的循环轮询模式，采用事件驱动唤醒机制，实现无流量时完全休眠，有流量时即时响应，大幅降低资源消耗。

2. **配置统一管理**：所有配置由监测服务统一管理，网关服务启动时通过类TCP三次握手机制同步配置，配置变更时自动通知同步，无需重启服务。

3. **状态机完善**：新增ATTACKING状态和MANUAL_RESET状态，完善状态转换条件和超时兜底机制，确保状态机流转正确可靠。

4. **防御后流量处理**：将被拦截流量纳入置信度计算和状态判定，修复防御后状态不再变化的严重问题。

5. **置信度科学计算**：基于多因素（频率、多样性、持续时间、攻击模式、历史行为等）科学计算置信度，采用只升不降策略避免状态反复横跳。

6. **流量采集优化**：采用事件驱动模式，实现轻量化采集、按需推送、队列容错、样本脱敏等功能。

通过以上优化，系统将具备更高的资源利用率、更强的防御能力、更好的可维护性和可扩展性。
