## 一、系统定位与架构概述

### 1.1 系统定位

本项目定位为**轻量级网络监测系统**，具有以下特点：

- **单网关部署**：与单一网关联动，无需集群设计
- **轻量化技术栈**：避免引入复杂组件，降低运维成本
- **核心功能聚焦**：专注于攻击检测、防御联动、告警通知

### 1.2 架构设计

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        轻量级混合架构                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   第一层防御（网关本地，实时）                                                │
│   ─────────────────────────────────────────────────────────────────────────  │
│   • 限流检测 ──> 本地计数，毫秒级响应                                        │

│   • 状态机 ──> NORMAL/SUSPICIOUS/ATTACKING/DEFENDED 本地判断                │
```

### 2.2 核心代码分析

#### 2.2.1 网关服务 - 状态机控制

**文件**: `gateway-service/src/main/java/com/network/gateway/cache/IpAttackStateCache.java`

```java
public StateTransitionResult checkAndTransitionState(String ip, int rateLimitCount, 
        int suspiciousThreshold, int attackingThreshold) {
    // 网关本地自动进行状态转换
    switch (currentState) {
        case NORMAL:
            if (rateLimitCount >= suspiciousThreshold) → SUSPICIOUS
        case SUSPICIOUS:
            if (rateLimitCount >= attackingThreshold) → ATTACKING
        case ATTACKING:
            if (持续攻击>30秒) → DEFENDED
        case DEFENDED:
            if (攻击停止) → COOLDOWN
        case COOLDOWN:
            if (冷却期结束) → NORMAL
    }
}
```

**分析**：

- ✅ 状态机逻辑完整，符合混合架构第一层防御要求
- ✅ 本地决策，实时响应

#### 2.2.2 网关服务 - 限流过滤器

**文件**: `gateway-service/src/main/java/com/network/gateway/filter/defense/RequestRateLimitFilter.java`

```java
public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
    // 1. 检查当前状态
    int currentState = attackStateCache.getState(sourceIp);
    
    // 2. DEFENDED状态直接拦截
    if (currentState == IpAttackStateConstant.DEFENDED) {
        return handleDefendedRequest(exchange, sourceIp, startTime);
    }
    
    // 3. 限流计数
    int currentRateLimitCount = attackStateCache.incrementRateLimitCount(sourceIp, windowMs);
    
    // 4. 状态转换
    StateTransitionResult transitionResult = 
        attackStateCache.checkAndTransitionState(sourceIp, currentRateLimitCount, ...);
    
    // 5. 状态转换后推送事件到监测服务
    if (transitionResult.isTransitioned()) {
        if (newState == ATTACKING) {
            pushDDoSEvent(sourceIp, ...);  // 推送DDoS事件
        } else if (newState == DEFENDED) {
            pushDDoSEvent(sourceIp, ...);
            pushBlacklistEvent(sourceIp, ...);  // 推送黑名单事件
        }
    }
}
```

**分析**：

- ✅ 实时限流检测
- ✅ 自动状态转换
- ✅ 事件推送到监测服务
- ⚠️ 推送事件后，监测服务可能再次生成防御指令（循环）

```java
public AttackMonitorDTO detect(TrafficMonitorDTO trafficDTO) {
    // 监测服务也有独立的DDoS检测逻辑！
    
    // 1. 独立的请求计数器

}
```

#### 2.2.4 监测服务 - 防御决策

**文件**: `monitor-service/src/main/java/com/network/monitor/service/impl/DefenseDecisionServiceImpl.java`

```java
public DefenseCommandDTO generateDefenseDecision(AttackMonitorDTO attackDTO) {
    // 1. 检查是否已防御
    if (attackStateCache.isInDefendedState(sourceIp)) {
        return null;  // 跳过
    }
    
    // 2. 根据风险等级生成防御指令
    if (riskLevel == HIGH || riskLevel == CRITICAL) {
        commandDTO = buildDefenseCommand(..., BLACKLIST, ...);
    }
    
    // 3. 推送防御指令到网关
    boolean success = gatewayApiClient.pushDefenseCommand(commandDTO);
    
    // 4. 记录防御日志
    recordDefenseLog(commandDTO, event, attackDTO);
    
    // 5. 发布黑名单同步事件
    publishBlacklistSyncEvent(commandDTO);
    
    // 6. 更新本地状态
    attackStateCache.markAsDefended(sourceIp, eventId);
}
```

**分析**：

- ✅ 完整的防御决策流程
- ✅ 记录防御日志和黑名单同步
- ⚠️ 网关已标记DEFENDED后，监测服务再推送指令（可能重复）

## 四、优化方案设计

### 4.1 优化目标

1. **消除重复检测**：DDoS检测逻辑统一由网关负责
2. **明确职责边界**：网关负责实时防御，监测服务负责分析记录
3. **简化通信机制**：单网关直接通信，无需广播机制

### 4.3 职责划分

| 层级      | 组件   | 职责                   | 决策权  |
| ------- | ---- | -------------------- | ---- |
| **第一层** | 网关服务 | 实时限流、黑名单拦截、状态转换、事件推送 | 本地决策 |
| **第二层** | 监测服务 | 事件聚合、数据存储、态势感知、告警通知  | 手动处置 |

### 4.4 具体优化项

#### 4.4.1 移除监测服务DDoS检测逻辑

**问题**: `DDoSDetectServiceImpl` 与网关限流检测逻辑重复

**方案**:

- 移除监测服务的独立DDoS检测
- 保留接口用于接收网关推送的攻击事件
- 监测服务只负责记录和分析，不再进行二次检测

**涉及文件**:

- `monitor-service/src/main/java/com/network/monitor/service/impl/DDoSDetectServiceImpl.java`
- `monitor-service/src/main/java/com/network/monitor/service/DDoSDetectService.java`

#### 4.4.2 优化事件处理流程

**问题**: 网关推送DDoS事件后，监测服务仍可能生成重复防御指令

**方案**:

```java
// AttackEventReceiveController.receiveDDoSEvent()
@PostMapping("/ddos-event")
public ApiResponse<Void> receiveDDoSEvent(@RequestBody Map<String, Object> eventData) {
    // 1. 检查IP是否已防御
    if (attackStateCache.isInDefendedState(sourceIp)) {
        log.info("IP已处于DEFENDED状态，仅记录事件: ip={}", sourceIp);
        // 仅保存攻击记录，不生成防御决策
        saveAttackRecord(eventData);
        return ApiResponse.success();
    }
    
    // 2. 保存攻击记录
    saveAttackRecord(eventData);
    
    // 3. 仅在需要时生成防御决策（如手动处置场景）
    // 自动防御由网关完成，此处不再重复
}
```

#### 4.4.3 优化防御日志记录

**问题**: 防御日志可能重复记录

**方案**: 增加去重逻辑，同一事件只记录一次

```java
// DefenseDecisionServiceImpl.recordDefenseLog()
private void recordDefenseLog(DefenseCommandDTO commandDTO, ...) {
    // 检查是否已记录
    if (defenseLogMapper.existsByEventId(commandDTO.getEventId())) {
        log.debug("防御日志已存在，跳过: eventId={}", commandDTO.getEventId());
        return;
    }
    
    // 记录防御日志
    DefenseLogEntity logEntity = new DefenseLogEntity();
    logEntity.setEventId(commandDTO.getEventId());
    logEntity.setIsFirst(1);
    // ... 其他字段
    defenseLogMapper.insert(logEntity);
}
```

## 五、流量采集与推送策略优化

### 5.1 当前策略分析

#### 5.1.1 过滤器执行顺序

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        过滤器执行顺序                                        │
├─────────────────────────────────────────────────────────────────────────────┤

│   0        IpBlacklistDefenseFilter         黑名单拦截                      │

│   10       RequestRateLimitFilter           限流检测                        │
│   20       MaliciousRequestBlockFilter      恶意请求拦截                    │
│                                                                             │
│   ✅ 流量采集过滤器优先级最高(-100)，会采集所有流量                           │

│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 5.1.2 当前推送策略

| IP状态       | 推送策略     | 说明              |
| ---------- | -------- | --------------- |
| NORMAL     | REALTIME | 实时推送每条流量        |
| SUSPICIOUS | SAMPLING | 采样推送（如每10条推送1条） |
| COOLDOWN   | SAMPLING | 采样推送            |
| ATTACKING  | BATCH    | 批量推送（定时聚合）      |
| DEFENDED   | BATCH    | 批量推送            |

#### 5.1.3 存在的问题

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        当前策略的问题                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   问题1: NORMAL状态实时推送压力大                                            │
│   ─────────────────────────────────────────────────────────────────────────  │
│   • 正常流量占绝大多数，实时推送会产生大量网络请求                             │
│   • 监测服务需要处理大量正常流量数据                                          │
│                                                                             │
│   问题2: 被拦截流量的推送策略不够优化                                         │
│   ─────────────────────────────────────────────────────────────────────────  │
│   • 黑名单拦截的流量（DEFENDED）使用BATCH，但可能仍占用大量带宽                │
│   • DDoS攻击时，被拦截的流量可能达到数万条/秒                                 │
│                                                                             │
│   问题3: 缺少对"被拦截流量"的特殊处理                                        │
│   ─────────────────────────────────────────────────────────────────────────  │
│   • 被拦截的流量有特殊标记（如状态码403/429）                                 │
│   • 这些流量的分析价值不同，应该有不同的推送策略                               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 5.2 优化后的推送策略

#### 5.2.1 核心原则

1. **完整采集**：所有流量都必须采集，包括被拦截的流量
2. **智能推送**：根据流量特征和系统负载动态调整推送策略
3. **数据聚合**：高频流量在网关侧聚合，减少推送量
4. **优先级区分**：异常流量优先推送，正常流量延迟推送

#### 5.2.2 优化后的策略定义

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        优化后的推送策略                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   流量类型              │ 推送策略        │ 说明                            │
│   ─────────────────────┼────────────────┼───────────────────────────────── │
│   正常流量(NORMAL)      │ DELAYED_BATCH  │ 延迟批量，每5秒聚合推送一次       │
│   可疑流量(SUSPICIOUS)  │ SAMPLING       │ 采样推送，每N条推送1条            │
│   攻击流量(ATTACKING)   │ AGGREGATE      │ 聚合推送，统计信息+关键样本       │
```

## 十三、总结

当前实现已经具备了轻量级监测系统的基础框架，但存在以下主要问题：

1. **双重DDoS检测**：网关和监测服务都有检测逻辑，造成资源浪费和潜在冲突
2. **流量推送效率低**：攻击时产生大量推送请求
3. **告警机制不完善**：缺少邮件/飞书通知、告警抑制等功能
4. **溯源能力有限**：缺少IP画像、攻击链可视化等

通过本次优化，将实现：

- 明确的职责边界：网关负责实时防御，监测服务负责分析记录
- 高效的流量推送：攻击时推送量减少99%以上
- 完善的告警机制：支持邮件/飞书通知、告警抑制、告警确认
- 增强的溯源能力：IP画像、攻击链可视化

优化后的架构将保持轻量级特点，具备更好的性能和可维护性。
