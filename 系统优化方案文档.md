## 一、问题概述

### 1.1 问题清单

| 编号 | 问题               | 影响                      | 严重程度 |
| -- | ---------------- | ----------------------- | ---- |
| P1 | IPv6地址无法匹配黑名单    | 攻击者使用IPv6地址可绕过黑名单防御     | 高    |
| P2 | DDoS攻击产生大量重复攻击记录 | 数据库膨胀、查询性能下降、日志可读性差     | 高    |
| P3 | 已防御IP仍走完整分析流程    | 系统资源浪费、响应延迟增加           | 高    |
| P4 | 防御指令重复推送         | 网络带宽浪费、网关缓存频繁更新         | 中    |
| P5 | 防御日志重复数据         | 防御日志表膨胀、统计不准确           | 中    |
| P6 | 前端不支持IPv6格式输入    | 无法手动添加IPv6地址到黑名单        | 中    |
| P7 | 态势感知能力不足         | 缺少聚合统计，需要SQL实时计算，查询性能差  | 中    |
| P8 | 防御效果评估缺失         | 缺少防御状态、持续时间等字段，无法评估防御效果 | 中    |

### 1.2 问题根因分析

#### 问题P1：IPv6地址无法匹配黑名单

**问题链路分析**：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        IPv6地址匹配失败问题链路                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. IP提取层（网关服务）                                                      │
│     ┌─────────────────────────────────────────────────────────────────┐     │
│     │ ServerWebExchangeUtil.extractSourceIp()                         │     │
│     │ remoteAddress.getAddress().getHostAddress()                     │     │
│     │ 本地请求返回: "0:0:0:0:0:0:0:1" (IPv6完整格式)                    │     │
│     └─────────────────────────────────────────────────────────────────┘     │
│                                    ↓                                        │
│  2. 黑名单存储层（监测服务）                                                   │
│     ┌─────────────────────────────────────────────────────────────────┐     │
│     │ BlacklistCache.add(ip, ...)                                     │     │
│     │ blacklistMap.put(ip, entry)  // 直接使用原始字符串作为key          │     │
│     │ 存储: "12.0.0.1" 或 "127.0.0.1"                                  │     │
│     └─────────────────────────────────────────────────────────────────┘     │
│                                    ↓                                        │
│  3. 黑名单查询层（网关服务）                                                   │
│     ┌─────────────────────────────────────────────────────────────────┐     │
│     │ IpBlacklistCache.isInBlacklist(ip)                              │     │
│     │ 查询: "0:0:0:0:0:0:0:1"                                          │     │
│     │ 结果: false (与存储的 "127.0.0.1" 不匹配)                          │     │
│     └─────────────────────────────────────────────────────────────────┘     │
│                                                                             │
│  根因: IP地址格式未统一规范化，导致同一地址的不同表示形式无法匹配              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 方案二：攻击状态标记机制（解决P2、P3、P4）

#### 状态定义

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          IP攻击状态定义                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  状态          │  说明                    │  后续处理                        │
│  ─────────────┼─────────────────────────┼───────────────────────────────── │
│  NORMAL        │  正常状态，无攻击行为     │  执行完整检测流程                 │
│  SUSPICIOUS    │  可疑状态，触发限流       │  加强监控，记录统计               │
│  ATTACKING     │  攻击中，已检测到攻击     │  执行防御，更新状态               │
│  DEFENDED      │  已防御，黑名单生效       │  直接拦截，不推送流量             │
```

### 2.3 方案三：两表分离设计（解决P2、P7、P8）

#### 设计思路

将攻击监测表和攻击事件表分离，各司其职：

| 表名                   | 职责               | 粒度           | 用途   |
| -------------------- | ---------------- | ------------ | ---- |
| sys\_attack\_monitor | 攻击追溯、规则优化、漏洞验证   | 每次关键攻击检测一条记录 | 详细分析 |
| sys\_attack\_event   | 态势感知、趋势分析、防御效果评估 | 每次攻击事件一条记录   | 聚合展示 |

#### 数据关系

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        两表数据关系                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  sys_attack_event（事件表）- 1条                                            │
│  ┌─────┬──────────┬───────────┬──────────────┬─────────────┬────────────┐  │
│  │ id  │ event_id  │ source_ip │ attack_type  │ total_requests│ peak_rps  │  │
│  ├─────┼──────────┼───────────┼──────────────┼─────────────┼────────────┤  │
│  │ 1   │ EVT-001   │ ::1       │ DDoS         │ 3000        │ 120        │  │
│  └─────┴──────────┴───────────┴──────────────┴─────────────┴────────────┘  │
│                          ▲                                                  │
│                          │ 关联（多对一）                                    │
│                          │                                                  │
│  sys_attack_monitor（攻击监测表）- 多条关键攻击记录                          │
│  ┌─────┬──────────┬────────────┬───────────┬───────────────┐              │
│  │ id  │ event_id  │ traffic_id │ confidence│ attack_content │              │
│  ├─────┼──────────┼────────────┼───────────┼───────────────┤              │
│  │ 101 │ EVT-001   │ TRF-1001   │ 92        │ 首次检测到DDoS │              │
│  │ 102 │ EVT-001   │ TRF-1089   │ 95        │ 攻击升级       │              │
│  │ 103 │ EVT-001   │ TRF-2156   │ 98        │ 峰值攻击       │              │
│  └─────┴──────────┴────────────┼───────────┴───────────────┘              │
│                                │                                            │
│                                └─ 仅记录关键攻击节点（首次、升级、峰值）        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 关键攻击节点定义

| 节点类型 | 触发条件                 | 是否记录  |
| ---- | -------------------- | ----- |
| 首次攻击 | 状态从NORMAL变为ATTACKING | ✅ 记录  |
| 攻击升级 | 置信度提升超过10%           | ✅ 记录  |
| 峰值攻击 | 请求频率达到峰值             | ✅ 记录  |
| 攻击结束 | 状态变为COOLDOWN         | ✅ 记录  |
| 普通攻击 | 已在ATTACKING状态        | ❌ 不记录 |

#### 新增 sys\_attack\_event 表结构


### 2.4 方案四：防御日志优化（解决P5）

#### 问题分析

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     防御日志重复问题                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  问题1: 限流日志重复                                                         │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ 当前: 每次限流都推送防御日志 → DDoS期间产生大量RATE_LIMIT日志          │   │
│  │ 优化: 限流日志关联事件ID，同一事件只记录首次限流和统计信息              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  问题2: 黑名单拦截日志重复                                                   │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ 当前: 已封禁IP的每次请求都产生BLOCK_IP日志                             │   │
│  │ 优化: DEFENDED状态的IP不推送防御日志，或仅推送统计信息                  │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  问题3: 黑名单事件重复                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │ 当前: 每条防御日志都可能发布黑名单事件                                  │   │
│  │ 优化: 检查IP是否已在黑名单中，避免重复添加                              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 改造 sys\_defense\_log 表

```sql
ALTER TABLE sys_defense_log 
ADD COLUMN `event_id` VARCHAR(64) COMMENT '关联事件ID' AFTER `id`,
ADD COLUMN `is_first` TINYINT DEFAULT 0 COMMENT '是否首次防御(0-否，1-是)' AFTER `execute_status`,
ADD INDEX `idx_event_id` (`event_id`);
```

#### 优化逻辑

| 场景      | 当前行为   | 优化后行为                       |
| ------- | ------ | --------------------------- |
| 首次限流    | 推送防御日志 | 推送防御日志，标记is\_first=1，关联事件ID |
| 后续限流    | 推送防御日志 | 不推送，仅更新事件统计                 |
| 首次黑名单拦截 | 推送防御日志 | 推送防御日志，标记is\_first=1，关联事件ID |
| 后续黑名单拦截 | 推送防御日志 | 不推送，或仅统计计数                  |
